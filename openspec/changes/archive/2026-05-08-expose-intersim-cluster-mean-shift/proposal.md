## Why

During implementation of `retune-toy-dataset-difficulty` we discovered that **the difficulty of `y = stage` PLS-DA classification on the toy dataset is not controllable by any flag currently exposed on `motco simulate`**. Stage classification difficulty is governed by InterSIM's per-omic cluster-mean separation parameters (`delta.methyl`, `delta.expr`, `delta.protein`), which position the cluster centroids that downstream code interprets as stages. With three well-separated centroids and ~660 omics features, supervised stage classification is trivially solvable regardless of `--effect-size` or `--prop-affected-features`.

The Python API already exposes these knobs through `InterSIMParams.delta_methyl`, `.delta_expr`, `.delta_protein` (`src/motco/simulations/intersim.py:62`), but the CLI does not. Exposing them on `motco simulate` would (a) let users dial CV-table difficulty for tutorials and demos, (b) enable empirical exploration of how cluster-separation interacts with the supervised PLS-DA latent space, and (c) unblock future Type-I and power studies (`src/simulations/motco_simulation_grid.py` references `cluster_mean_shift` as a Phase-1 OFAT axis).

## What Changes

- **`src/motco/cli.py`** — extend the `simulate` subparser with three new optional flags:
  - `--delta-methyl` (float, default `None` → use InterSIM default): per-feature mean shift between clusters for methylation.
  - `--delta-expr` (float, default `None`): per-feature mean shift for expression.
  - `--delta-protein` (float, default `None`): per-feature mean shift for proteomics.
  - Plus a convenience flag `--cluster-mean-shift` (float, default `None`): when set, fills any of the three per-omic deltas that were not individually specified. Mutually compatible with the per-omic flags (per-omic always wins).
  - Each value, if provided, must be a non-negative float; out-of-range exits before invoking R.
- **`cmd_simulate`** — thread the resolved deltas into `InterSIMParams`.
- **Tests** — extend `tests/test_cli.py` so each flag is wired through to `InterSIMParams` (mocked) and that `--cluster-mean-shift` fills only the missing per-omic deltas.
- **Docs** — small update to `simulate-command` capability spec; update `README.md` and the notebook only if either currently quotes a delta value (they don't today).

## Capabilities

### Modified Capabilities

- `simulate-command`: adds four optional CLI flags (`--delta-methyl`, `--delta-expr`, `--delta-protein`, `--cluster-mean-shift`); existing flags and defaults are unchanged. Validation: each, if provided, must be `>= 0`.

### New Capabilities

(none)

## Impact

- `src/motco/cli.py`: new flags + small block in `cmd_simulate` that resolves per-omic deltas from `--cluster-mean-shift` when individual flags are absent.
- `tests/test_cli.py`: 2–3 new tests.
- `openspec/specs/simulate-command/spec.md`: updated requirements (covered by spec deltas).
- `examples/data/toy/`: NOT regenerated by this change. The toy demonstration is not a fit for showcasing a cluster-mean-shift sweep; if a future change wants a "harder tutorial" toy variant, that's a separate proposal.
- No external dependencies added or removed.
- Optional R + InterSIM runtime dependency for `motco simulate` is unchanged. New flags are forwarded to InterSIM's existing parameters; no R-side change is needed.

## Open Questions

- **Do we want a `--cluster-mean-shift` convenience flag, or only the three per-omic flags?** The convenience flag is friendlier for tutorials; the per-omic flags are more honest about the underlying parametrization. Current proposal: ship both; per-omic wins on conflict. If the user prefers strictly one or the other, the design.md decisions will adjust.
- **Should `--cluster-mean-shift` accept a list of three values (one per omic) instead of a single scalar?** Argparse can do `nargs=3`; would let users pass `--cluster-mean-shift 1.0 0.8 1.2`. Adds CLI complexity; current proposal sticks with a scalar that fans out, leaving per-omic overrides for the dedicated flags.
